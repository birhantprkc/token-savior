# Migrating to Token Savior v3.0

Short version: **v3.0 flips the default profile from `full` (94 tools) to
`lean` (61 tools).**

## Quick rollback (3 lines)

```bash
export TOKEN_SAVIOR_PROFILE=full   # restore v2.x default
# or: pip install 'token-savior-recall>=2.8,<3'   # stay on v2 line
```

That's it — no schema migration, no data loss, cache carries over.

v2.8 keeps the old default but prints a one-line deprecation warning at
server boot so the change does not surprise anyone when v3.0 lands.

## What changes

| | v2.8 (current) | v3.0 (breaking) |
|---|---|---|
| Default `TOKEN_SAVIOR_PROFILE` | `full` | `lean` |
| Tools advertised at handshake | 94 (~16 000 tokens) | 61 (~11 700 tokens) |
| Memory engine tools visible | yes | **no** (except `memory_save`, which stays) |
| Reasoning / corpus tools visible | yes | **no** (opt-in) |
| Niche analysis tools (`get_duplicate_classes`, `get_call_predictions`, `pack_context`) | yes | **no** (opt-in) |
| `discover_project_actions` + `run_project_action` | yes | **yes** (kept as atomic pair) |

Tools are never removed, only filtered from `list_tools`. Every dropped
tool can be called by explicit name, or re-advertised by setting the
env var.

## Why

Empirical post-v2.8.1 measurements:

- `full`  at **~15 986 tokens** at MCP handshake reliably triggers
  Claude Code's 10k auto-defer threshold.
- `lean`  at **~11 663 tokens** may trigger auto-defer.
- `ultra` at **~3 852 tokens** stays fully visible on any client.

Anthropic's engineering research ([Advanced tool use](https://www.anthropic.com/engineering/advanced-tool-use),
cited in AUDIT.md Phase 3.6) notes tool-selection accuracy degrades
significantly past 30–50 visible tools. 61 is already on the high end;
94 was too many.

**Why `memory_save` stays in the default**: the user-facing
"remember this across sessions" path is the core value proposition of
Token Savior Recall. Dropping it silently would break the README's
"nothing forgotten" contract.

**Why `discover_project_actions` + `run_project_action` stay**: they
work as an atomic pair (discover, then run). Dropping only one breaks
the workflow. Spike-1 measurements showed they see 5/3330 VPS calls
(low but non-zero), paired usage.

## Three ways to keep the v2.8 behaviour on v3.0+

1. **Set the env var** on the server side:
   ```bash
   export TOKEN_SAVIOR_PROFILE=full
   ```

2. **Tell your MCP client to keep everything hot** — on Anthropic API,
   use `mcp_toolset` with `default_config.defer_loading: false`. See
   `docs/mcp_toolset.example.json` for the recommended layout (17
   non-deferred hot tools + 77 deferred via Anthropic Tool Search).

3. **Pin to the v2.x line**:
   ```toml
   token-savior-recall = ">=2.8,<3"
   ```

## If you relied on memory tools visible by default

`memory_save` stays in `lean`. The 25 other `memory_*` tools
(`memory_search`, `memory_get`, `memory_index`, etc.) are still
callable by exact name — handlers stay wired — they just do not appear
in `list_tools`. The auto-injection hooks (`hooks/memory-*.sh`) talk
to the database directly, not through MCP tools, so passive memory
capture and recall are unaffected by the profile change.

## What v3.0 does *not* change

- Tool names or signatures.
- Memory schema.
- Hook lifecycle.
- Cache format.
- Python version support (>= 3.11).
- Nomic embedding model.
- Behaviour when the user sets `TOKEN_SAVIOR_PROFILE` explicitly.
